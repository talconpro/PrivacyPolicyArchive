# 任务 09：完善风险标签与评分引擎

## 目标
将当前“模型直接输出风险等级”的做法，升级为“结构化因子 + 规则计算 + 等级映射”的可解释评分系统。

## 需要 Codex 完成

1. 新增风险评分模块，例如：`src/lib/risk/score.ts`
2. 定义统一的风险输入结构，例如：
   - `dataCollectionFlags`
   - `dataSharingFlags`
   - `userRightsFlags`
   - `contractFlags`
   - `transparencyFlags`
   - `minorProtectionFlags`
3. 实现：
   - `calculateRiskScore(analysis)`
   - `mapRiskLevel(score)`
   - `applyHardTriggers(analysis, currentLevel)`
4. 将评分结果写回数据库字段：
   - `riskScore`
   - `riskLevel`
   - `riskReasons`
5. 在详情页展示：
   - 综合风险等级
   - 风险分数
   - 主要触发原因（3 条以内）

## 验收标准

- 相同分析输入可稳定产出相同分数与等级
- 高风险命中项可解释
- 前端页面可以看到“为什么是这个等级”

## 推荐给 Codex 的提示词

```md
请在当前项目中实现“风险评分引擎”，要求：
1. 不再只依赖 LLM 直接给出的 riskLevel。
2. 基于 analysis JSON 中的结构化字段计算 riskScore，并映射为 low / medium / high / severe。
3. 支持硬触发规则，例如：出售个人数据、无限制第三方共享、强制仲裁等至少将等级提升为 high。
4. 输出 TypeScript 类型定义、纯函数实现、单元测试，并接入现有分析落库流程。
5. 在详情页增加“评分依据”展示。
请先阅读现有 Prisma schema、分析模块和详情页组件后再修改。
```

# 任务 10：补全分析输出 Schema 与校验

## 目标
让 AI 分析结果强制遵循统一 JSON 结构，避免脏数据污染页面与数据库。

## 需要 Codex 完成

1. 定义 analysis schema，可用 `zod`：
   - `oneLineSummary`
   - `plainSummary`
   - `keyFindings`
   - `dataCollected`
   - `dataShared`
   - `userRights`
   - `redFlags`
   - `confidence`
   - `needsHumanReview`
2. 在分析模块加入：
   - JSON 解析失败处理
   - schema 校验失败处理
   - fallback 默认值
3. 落库前统一经过 `normalizeAnalysisResult()`
4. 将错误日志写入分析任务日志中

## 验收标准

- 非法 JSON 不会导致任务整体中断
- 缺字段时可安全降级
- 页面不再直接依赖不可信原始字段

## 推荐给 Codex 的提示词

```md
请为 AI 分析结果增加严格的 schema 校验与标准化处理：
- 使用 zod 定义 schema
- 在分析模块里完成 parse -> validate -> normalize -> persist 流程
- 保证异常情况下不会写入不可用数据
- 为常见失败场景补测试
- 尽量少改现有调用方式
```

# 任务 11：加入人工复核队列

## 目标
将高风险、低置信度、异常文本等结果标记出来，供人工抽检。

## 需要 Codex 完成

1. Prisma schema 增加复核字段或新表，例如：
   - `reviewStatus`
   - `reviewReason`
   - `reviewPriority`
2. 规则：以下情况自动进入复核队列
   - `riskLevel = severe`
   - `confidence < 0.7`
   - 原文过短或过长异常
   - 命中“出售数据/强制仲裁/永久保留”等高敏感标记
3. 新增一个后台页或本地脚本用于查看待复核列表
4. 支持人工标记：
   - approved
   - corrected
   - false_positive

## 验收标准

- 复核数据可查询
- 高风险结果能自动入队
- 不影响静态页生成

## 推荐给 Codex 的提示词

```md
请在现有项目中实现“人工复核队列”能力：
- 为 analysis 或独立 review queue 增加数据结构
- 在分析完成后自动判断是否需要人工复核
- 提供一个最小可用的管理界面或 CLI 列表命令
- 优先保证实现简单、可维护
```

# 任务 12：完善协议版本化与历史记录

## 目标
支持同一 App 多个协议版本的存储与切换，为后续变更追踪打基础。

## 需要 Codex 完成

1. 检查并补全 `PolicyVersion` 设计：
   - `appId`
   - `documentType`
   - `sourceUrl`
   - `rawText`
   - `contentHash`
   - `fetchedAt`
   - `isCurrent`
2. 更新抓取流程：
   - 若 `contentHash` 未变化则不创建新版本
   - 若变化则新增版本，并将旧版本 `isCurrent=false`
3. 在详情页加入“历史版本”入口
4. 新增历史版本列表页或详情区块

## 验收标准

- 同一 App 至少可看到多个历史版本
- 当前版本标记正确
- 不会重复生成相同 hash 的版本

## 推荐给 Codex 的提示词

```md
请完善协议版本化：
- 基于 contentHash 判断是否产生新版本
- 保证 current version 唯一
- 在 App 详情页增加历史版本入口
- 最小实现即可，不需要做复杂 diff UI
```

# 任务 13：实现“变更摘要”而不是复杂 diff

## 目标
对协议新旧版本生成用户可读的一句话变更摘要，先不做精细逐句比对。

## 需要 Codex 完成

1. 在版本更新后，比较“当前版本 vs 上一版本”
2. 调用 LLM 生成：
   - `changeSummary`
   - `changeHighlights[]`
3. 只输出高价值变更，例如：
   - 新增第三方共享描述
   - 扩大位置数据用途
   - 补充删除数据路径
4. 在页面展示“最近变更”模块

## 验收标准

- 页面能看到最近一次变更摘要
- 没有上一个版本时自动跳过
- 失败不阻塞主流程

## 推荐给 Codex 的提示词

```md
请实现协议版本的“变更摘要”功能：
- 对比相邻两个版本原文
- 使用 LLM 生成一句话摘要和 3 条以内 highlights
- 不做逐行 diff，只做用户可读的总结
- 接入现有落库与详情页展示流程
```
